## ADDED Requirements

### Requirement: Live component render harness
The Playground SHALL render the **real** generated component (not a re-implementation or an IR render) inside the app, by launching the project's browsable surface (a dev server exposing a gallery route, or Storybook) via the managed PTY and embedding its URL in a webview. VortSpec SHALL NOT implement its own per-framework renderer or bundler.

#### Scenario: Real component renders in an embedded preview
- **WHEN** the user opens the Playground for the Button component and a browsable surface is available
- **THEN** VortSpec SHALL launch it via the managed PTY and embed its URL in a webview showing the real Button as rendered by the project's own stack

#### Scenario: Raw surface is always reachable
- **WHEN** the Playground preview is shown
- **THEN** an "open in browser" escape hatch and the terminal/log view SHALL be available (transparency)

### Requirement: Harness generation via Claude Code when absent
When the project has no browsable surface (no Storybook, no gallery route — the current `visual-verify` blocker), the Playground SHALL offer to generate one by running a scoped Claude Code step that writes framework-correct harness files (a minimal gallery route or Storybook stories) into the project. VortSpec SHALL NOT write framework renderer code itself; the engine (Claude Code) writes it, targeting the project's actual framework.

#### Scenario: Missing harness is generated by the engine, gated
- **WHEN** the user opens the Playground and no renderable surface exists
- **THEN** VortSpec SHALL offer to generate a harness via a Claude Code run
- **AND** the generated harness files SHALL be shown to the user and written into the project transparently (git-ignorable), never silently

#### Scenario: Harness generation closes the visual-verify gap
- **WHEN** a harness has been generated and the dev server is running
- **THEN** the `visual-verify` stage's live-viewport and axe checks SHALL become executable against that surface

### Requirement: Variant and state matrix rendering
The Playground SHALL render each component across its variants and states, driving them from controls generated from the component's source-declared props (variant/enum → select, boolean → toggle, string → text input). Changing a control SHALL update the live preview.

#### Scenario: Variant switching updates the live preview
- **WHEN** the user selects "secondary" in the Button's variant control
- **THEN** the embedded preview SHALL re-render the real secondary Button
- **AND** the control SHALL highlight "secondary" as active

#### Scenario: State rendering
- **WHEN** a component declares states (e.g. hover, disabled)
- **THEN** the Playground SHALL render a preview for each declared state

### Requirement: Validate and request modifications (gated)
The Playground SHALL surface validation results for the previewed component by reusing the existing `visual-verify` / `adversarial-review` report outputs (untokenized values, missing states, a11y gaps), and SHALL let the user request a fix. Requested fixes SHALL route through Claude Code (a scoped run or the resumable chat) and be applied only after the user approves the resulting diff (spec-first gate). No component code is mutated silently or by VortSpec directly.

#### Scenario: Issues shown from verify reports
- **WHEN** the previewed component has an open discrepancy in its `visual-verify` report
- **THEN** the Playground SHALL display that issue alongside the preview

#### Scenario: Requested fix is gated through the engine
- **WHEN** the user requests a change to the previewed component
- **THEN** VortSpec SHALL run a scoped Claude Code step and present the proposed change for approval
- **AND** the change SHALL be written to the project only after the user approves it

### Requirement: Local-first and framework-agnostic
The Playground SHALL derive everything it renders from the project's own files and stack, work regardless of the project's framework (React/Vue/Svelte/…), and require no VortSpec account, provider key, or network call beyond the local dev server. Preview infrastructure (PTY, webview) SHALL be isolated so non-macOS support remains a contained change.

#### Scenario: Works for a non-React project
- **WHEN** the project's framework is Vue (per `.sdd-de/project.yaml`)
- **THEN** harness generation and preview SHALL target Vue, with no VortSpec-authored renderer involved

#### Scenario: No credentials required
- **WHEN** the Playground launches a preview
- **THEN** it SHALL require no VortSpec account or provider key and make no network calls other than to the local dev server
