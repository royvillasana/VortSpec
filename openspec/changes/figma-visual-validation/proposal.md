## Why

Components generated by the current SDD-DE build no longer resemble their Figma designs — an "alert" comes out looking like a copy of a button, and many components are visibly wrong. This is a **regression**: an earlier approach produced better-matching components. The root cause is that the build lost its design anchor: `buildOnePrompt` implements a component from **only its name + the extracted tokens** (*"Implement the 'alert' component… using ONLY the extracted design tokens"*) — it never looks at what the component actually looks like in Figma, so the agent invents a plausible shape from the word alone. And nothing validates the result: the "visual-verify" step greps for hardcoded values and (as of the recent compile-gate change) typechecks, but it never renders the component and compares it to the design — and it *couldn't*, because the Figma file exposes only Cover/Typography/Icons pages, with no per-component frames to compare against. So low-fidelity components sail through as "verified."

The user's Figma convention closes both gaps: **each Figma page is one component and contains that component with its variations** (page "accordion" → the accordion + all its variants). That gives every component an authoritative, discoverable visual reference — the anchor the build needs and the target the validation compares against.

## What Changes

- **Figma page-per-component convention**: treat each Figma page named for a component as the authoritative reference for that component (the component + its variant set). VortSpec discovers these pages, maps each to a roster component, and retrieves its design (structure, variant frames) and a screenshot.
- **Design-anchored build**: the build/spec-generation reads the component's Figma reference (design + screenshot + variant frames) and builds to match it, instead of inventing from name + tokens. **BREAKING** to the current name-only `buildOnePrompt` behavior.
- **AI visual-validation gate**: before a component is accepted, the agent renders it (Storybook), screenshots each variant/state, and compares the render to the component's Figma reference, producing a fidelity verdict. Validation runs in a fixed order — **visual → token → code** — and a component is not "verified" until the visual layer passes (or a real, listed reason blocks it). No green PASS on fidelity the AI did not actually compare.
- **Regression root-cause captured**: document current-vs-previous so the design anchor that made the earlier approach better is restored deliberately, not by accident.

## Capabilities

### New Capabilities
- `figma-component-reference`: the page-per-component Figma convention and the per-component design retrieval (map a roster component → its Figma page; fetch its structure, variant frames, and screenshot) that serves as the authoritative reference for both building and validating.
- `visual-validation-gate`: the ordered visual → token → code validation pipeline that renders a built component, compares it to its Figma reference, and gates acceptance on real visual fidelity (never a self-certified pass).
- `design-anchored-build`: building/spec-generating a component from its Figma reference (design + screenshot) rather than from its name + tokens alone.

- The SDD-DE methodology docs (`@royvillasana/sdd-de`) — the page-per-component authoring convention.
- Depends on a reachable Figma bridge/MCP and a running Storybook (or dev preview) for the render-and-compare loop.
