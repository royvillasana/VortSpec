Troubleshooting & First-Run Reference

When something goes wrong

Exactly what the IDE checks, what it can install for you, what can actually stop you, and how to clear each issue. Errors in the app are always shown as fix-it cards with a next step — this page is the deeper reference.

What we validate on first run

The environment scan is read-only — it reports status and offers a fix, and never changes anything on its own.

CheckHowRequired?If it fails
Node.jsnode --version, must be ≥ 20BlockerInstall / update link
Gitgit --versionBlockerInstall link
Claude Codeclaude --versionBlockerInstall link
Claude loginOne-word headless probe, only when you click VerifyBlocker to run"Open login"
Figma MCPclaude mcp list (no usage)OptionalInfo only — set up if you want Figma

The login check spends a tiny amount of your Claude usage, so it only runs on request — never in the routine scan.

What VortSpec installs — only with your permission

VortSpec never installs system tools silently. On an explicit action it can set up:

  • Project dependencies — npm install automatically the first time you Run a project that has no node_modules.
  • A .env file from your project's .env.example — a one-click helper. It never fabricates secret values; you fill those in.
  • figma-cli — installed and connected for you if you choose to use Figma.
  • The SDD-DE toolkit — scaffolded into a new project (npx @royvillasana/sdd-de).

System tools it depends on (Node, Git, Claude Code) are never auto-installed — it points you to the official download instead.

What counts as a blocker

A blocker is anything that prevents the IDE from doing its core job. There are only a few, and each has a one-click fix:

BlockerWhy it stops youFix
Node.js missing / < 20Can't run installs or your dev server / previewInstall Node 20+ from the link, then re-check
Git missingNo Source Control, clone, or historyInstall Git, then re-check
Claude Code not installedNo engine — the assistant and all runs need itInstall Claude Code, then re-check
Claude Code not logged inRuns fail with an authentication error"Open login", sign in, then Verify
App won't launch (Gatekeeper)Unsigned build blocked on first openSee below
Not a blockerA missing or unconnected Figma connection only disables Figma design sources. Everything else in the IDE works without it.

The app won't open (Gatekeeper) — incl. "is damaged"

Either "VortSpec IDE can't be opened because Apple cannot check it for malicious software" or — on Apple Silicon — "VortSpec IDE is damaged and can't be opened. You should move it to the Trash." The build isn't notarized yet; despite the wording, the app is not actually damaged.

Both come from the same cause (a downloaded, not-yet-notarized app). After dragging the app to Applications, clear the quarantine flag once — this fixes both messages:

xattr -cr "/Applications/VortSpec IDE.app"

Then open it normally. For the milder "cannot check…" wording you can instead right-click the app → Open → Open, or System Settings → Privacy & Security → Open Anyway. (The "is damaged" wording usually needs the xattr command above.) You only need to do this once per install.

NoteIf you already moved it to the Trash, put it back in Applications first, then run the command. The latest builds are ad-hoc-signed so the harsher "is damaged" message shouldn't appear on a fresh download.

"npm is not installed or not on your PATH"

The dev server or Storybook exits immediately with "npm is not installed or not on your PATH", and the Run Doctor keeps looping even though Node/npm are installed.

This was a bug in older builds: an app launched from Finder gets a minimal PATH, and the fix arrived a moment too late for the dev server. It's resolved in the latest build.

FixUpdate to the latest IDE build and Run again. (If you use a Node version manager like nvm, the latest build finds it correctly.)

Missing environment variables / secrets

The preview fails with something like "supabaseUrl is required" or another missing-config error — common right after cloning a repo.

The project needs a .env you don't have yet. In the Run view, VortSpec detects this and offers a helper to create .env from the project's .env.example. Then fill in the real values (API URLs, keys) — VortSpec never invents secrets for you.

Heads-upIf the .env.example has placeholder values (like <your-key>), the app flags them so you know what still needs a real value.

The preview / dev server won't start

"Preview process exits code 1", or the dev server errors out.

Work through it in order:

  1. VortSpec surfaces the real error tail, not a bare exit code — read the last lines.
  2. A freshly cloned repo installs its dependencies automatically on first Run; let that finish.
  3. If it's a missing .env, use the helper above.
  4. Otherwise use the Run Doctor — a one-click, gated diagnosis that uses Claude Code to find the cause and propose a fix. It never fabricates secrets; if a value is genuinely required, it tells you what to provide.

macOS asked for a permission that seems too broad

macOS prompts to access something broad — e.g. your Apple Music library or other folders — when the assistant runs.

VortSpec declares no such permissions and has no interest in your music. This is a macOS runtime prompt that appears when a tool it runs (the assistant) touches a protected folder. It is always safe to click "Don't Allow" — the IDE works fine without it.

Fixed in the latest buildThe assistant now runs in a dedicated, scoped folder instead of your home directory, and the file watcher no longer spans protected areas — so these prompts no longer appear. The unused camera/microphone/Bluetooth declarations were also removed. Update to get this.

The assistant isn't responding

Chats do nothing, or runs fail instantly.

  • Confirm Claude Code is installed (claude --version in a terminal) and on your PATH.
  • Confirm you're logged in — run claude once in a terminal and sign in, or use the "Open login" fix-it card, then Verify.
  • VortSpec uses your own login (never a bare API key), so no keys or accounts are needed on the VortSpec side.

Reset & where your data lives

  • Your projects are plain files in the folders you chose — nothing is hidden inside the app.
  • App settings & recent-projects list live in ~/Library/Application Support/VortSpec IDE.
  • Your Claude login, plan, and usage belong to your own Claude Code install — VortSpec stores no keys and sends no telemetry.
  • To start clean, close the app and remove that Application Support folder; your project files are untouched.
